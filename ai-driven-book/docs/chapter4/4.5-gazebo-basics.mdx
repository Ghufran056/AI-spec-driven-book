# Gazebo Basics

## Learning Objectives

By the end of this lesson, you will be able to:
- Understand the fundamental concepts and architecture of the Gazebo simulation environment
- Navigate the Gazebo user interface and understand its main components
- Create and modify simple simulation worlds using Gazebo
- Spawn and control robot models in Gazebo simulation
- Configure basic physics properties and sensor models in Gazebo
- Understand how to integrate Gazebo with ROS 2 for robot simulation

## Introduction

Gazebo is a powerful, open-source 3D simulation environment widely used in robotics research and development. It provides realistic physics simulation, high-quality graphics, and support for various robot models and sensors. Gazebo serves as a bridge between theoretical robot algorithms and real-world implementation, allowing developers to test and validate their systems in a safe, controlled environment before deploying on physical robots.

The simulation environment is built on robust physics engines like ODE, Bullet, and DART, providing accurate modeling of real-world physics including collision detection, friction, and dynamics. Gazebo integrates seamlessly with ROS and ROS 2, making it an essential tool in the robotics development pipeline. Understanding Gazebo basics is crucial for anyone working with robot simulation, as it provides the foundation for testing navigation algorithms, sensor fusion, manipulation strategies, and other complex robotic behaviors.

## Main Theory

### 1. Gazebo Architecture
Gazebo consists of three main components: the physics engine, the rendering engine, and the sensor engine. These components work together to provide realistic simulation of robot behavior in virtual environments.

### 2. World Description Format
Gazebo uses SDF (Simulation Description Format) to define simulation worlds, including robot models, environment objects, lighting, and physics properties. SDF files describe the entire simulation environment in XML format.

### 3. Physics Simulation
Gazebo's physics engine accurately models real-world physics including gravity, friction, collision detection, and dynamic responses. It supports multiple physics engines to suit different simulation requirements.

### 4. Sensor Simulation
Gazebo includes realistic sensor simulation for cameras, LIDAR, IMU, GPS, and other sensor types. Each sensor model includes noise characteristics and physical limitations that match real hardware.

### 5. Control Interface
Gazebo provides interfaces for controlling simulated robots, including direct model manipulation, joint control, and integration with ROS/ROS 2 control systems.

### 6. Plugin System
Gazebo supports a plugin architecture that allows extending functionality through custom plugins for controllers, sensors, and other simulation elements.

```
Gazebo Architecture:

[User Interface] ←→ [Gazebo Server] ←→ [ROS 2 Bridge]
     ↑                  ↑                  ↑
[Simulation] ←→ [Physics Engine] ←→ [Control Systems]
     ↑                  ↑                  ↑
[Rendering] ←→ [Sensor Models] ←→ [Algorithms]
```

## Examples

### Example: Basic Gazebo Launch Command
```bash
# Launch Gazebo with an empty world
gazebo

# Launch Gazebo with a specific world file
gazebo my_world.world

# Launch Gazebo with GUI disabled (for headless simulation)
gzserver my_world.world
```

### Example: Simple World File
```xml
<?xml version="1.0" ?>
<sdf version="1.7">
  <world name="basic_world">
    <!-- Physics engine configuration -->
    <physics type="ode">
      <gravity>0 0 -9.8</gravity>
      <max_step_size>0.001</max_step_size>
      <real_time_factor>1</real_time_factor>
    </physics>

    <!-- Lighting -->
    <light name="sun" type="directional">
      <pose>0 0 10 0 0 0</pose>
      <diffuse>1 1 1 1</diffuse>
      <specular>1 1 1 1</specular>
      <attenuation>
        <range>1000</range>
        <constant>0.9</constant>
        <linear>0.01</linear>
        <quadratic>0.001</quadratic>
      </attenuation>
    </light>

    <!-- Ground plane -->
    <model name="ground_plane">
      <static>true</static>
      <link name="link">
        <collision name="collision">
          <geometry>
            <plane><normal>0 0 1</normal></plane>
          </geometry>
        </collision>
        <visual name="visual">
          <geometry>
            <plane><normal>0 0 1</normal></plane>
          </geometry>
          <material>
            <ambient>0.8 0.8 0.8 1</ambient>
            <diffuse>0.8 0.8 0.8 1</diffuse>
          </material>
        </visual>
      </link>
    </model>

    <!-- Simple robot model -->
    <model name="simple_robot">
      <pose>0 0 0.5 0 0 0</pose>
      <link name="chassis">
        <inertial>
          <mass>1.0</mass>
          <inertia>
            <ixx>0.1</ixx>
            <ixy>0</ixy>
            <ixz>0</ixz>
            <iyy>0.1</iyy>
            <iyz>0</iyz>
            <izz>0.1</izz>
          </inertia>
        </inertial>
        <collision name="collision">
          <geometry>
            <box><size>1 0.5 0.3</size></box>
          </geometry>
        </collision>
        <visual name="visual">
          <geometry>
            <box><size>1 0.5 0.3</size></box>
          </geometry>
          <material>
            <ambient>0.5 0.5 1 1</ambient>
            <diffuse>0.5 0.5 1 1</diffuse>
          </material>
        </visual>
      </link>
    </model>
  </world>
</sdf>
```

### Example: Spawning a Model in Gazebo
```bash
# Spawn a model from a file
ros2 run gazebo_ros spawn_entity.py -file my_robot.sdf -entity my_robot -x 0 -y 0 -z 1

# Spawn a model from a parameter server
ros2 run gazebo_ros spawn_entity.py -entity my_robot -database turtlebot3_waffle
```

## Practical Notes

- Always test simulation with simple models first before moving to complex robots
- Configure physics parameters appropriately for your specific use case
- Use appropriate update rates for sensors to balance accuracy and performance
- Regularly save your world files to avoid losing work
- Test simulation behavior against real-world expectations
- Consider computational resources when designing complex simulation scenarios
- Use Gazebo's built-in models when possible to save development time

## Summary

Gazebo provides a comprehensive simulation environment for robotics development, offering realistic physics simulation, sensor modeling, and integration with ROS/ROS 2. Understanding Gazebo basics is essential for effective robot simulation, allowing developers to test algorithms safely and efficiently before deployment on physical robots. The combination of SDF world descriptions, physics engines, and sensor models makes Gazebo a powerful tool for robotics research and development.

## Glossary

- **Gazebo**: Open-source 3D simulation environment for robotics research and development
- **SDF**: Simulation Description Format - XML-based format for describing simulation environments
- **Physics Engine**: Software component that calculates realistic physical interactions in simulation
- **World File**: SDF file that describes an entire simulation environment including models, lighting, and physics
- **ROS Bridge**: Interface that connects Gazebo simulation with ROS/ROS 2 systems
- **Sensor Simulation**: Modeling of real-world sensors in virtual environments with realistic noise and limitations
- **Model**: 3D representation of a physical object in the simulation environment
- **Plugin**: Extension code that adds functionality to Gazebo simulation
- **Collision Detection**: Computational method for identifying when objects intersect in simulation
- **Update Rate**: Frequency at which simulation calculations are performed
- **Real Time Factor**: Ratio of simulation time to real time (1.0 = real-time simulation)
- **Joint Control**: Method for controlling robot joint positions and movements in simulation
